Introdução
02. Os elementos da Skill
Uma Skill é um mini-manual que o Project consulta para tarefas específicas

Uma Skill do Claude é um mini-manual que você instala no Project. Quando você anexa uma NF e pede “extrai isso”, o Claude detecta que existe uma Skill para esse tipo de tarefa e segue o roteiro descrito nela — em vez de improvisar.

1
Input definido
A Skill declara que aceita NF em PDF, imagem ou XML. Qualquer outro formato vira recusa.
2
Passos numerados
A Skill segue 4 passos fixos: receber a NF → identificar campos visualmente → estruturar em JSON → verificar consistência.
3
Output em schema fechado
A Skill devolve sempre o MESMO formato JSON, com os mesmos campos, na mesma ordem. Sem variação criativa.

Nesta página, a Skill já está pronta para download em formato .zip. Você vai instalar no Project que criou na AP01 e testar com 2 NFs que também já estão prontas.

Introdução
03. O JSON de saída — schema enxuto
Apenas 6 campos no topo — e por que isso importa

A Skill devolve um JSON com apenas 6 campos no topo (mais alguns nested). Esse schema enxuto foi desenhado para:

3 razões para schema enxuto

  • Minimizar tokens consumidos em cada chamada da Skill
  • Conter exatamente o que a AP03 precisa para cruzar a NF contra o pedido
  • Eliminar campos supérfluos que não serão usados nas APs seguintes
Evite — schema inflado
30+ campos: número, série, modelo, natureza da operação, CFOP, IE, fornecedor completo, destinatário completo, todos os impostos, frete, seguro, modalidade, peso, volume, dados bancários, observações...
Cada campo extra é token gasto à toa. A AP03 vai usar 6 desses 30 — o resto só engrossa a janela de contexto e degrada performance.
Use — schema enxuto
6 campos: numero_nf, data_emissao, fornecedor (razão social + CNPJ), valor_total, itens (descrição + qtd + valor unitário)
Exatamente o que a Skill de conciliação precisa. Cada campo justifica sua existência no pipeline.
Princípio de schema mínimo

Sempre que você desenhar uma Skill que devolve dados estruturados, comece pelo consumo a jusante (quem vai usar esse JSON?) e adicione SÓ os campos necessários. Schema cresce; não nasce grande. É mais fácil adicionar um campo depois do que tirar.

Introdução
04. A Skill extrai. A NF pode mentir.
O princípio “LLM lê, código calcula” funcionando na prática

Aqui está o detalhe crítico desta AP, que vai te preparar para a AP03:

A Skill extrai os valores que aparecem IMPRESSOS na NF — ela não recalcula nada. Se o fornecedor errou a soma ao emitir a NF (e isso acontece muito mais do que você imagina), o JSON vai sair com o erro do fornecedor. A descoberta da inconsistência só acontece quando uma OUTRA camada (code execution na AP03) reconfere os valores.

Esse é o princípio “LLM extrai, código calcula” funcionando na prática.

Pedir pro LLM corrigir
“Se a soma dos itens não bate com o total da NF, recalcule e devolva o valor correto.”
A Skill perderia a informação original do erro. O analista nunca saberia que o fornecedor errou. Você passaria a confiar cegamente no Claude para calcular — voltando ao problema original.
Extrair fiel, validar depois
“Devolve o JSON com os valores como aparecem na NF. Mesmo que pareça inconsistente, NÃO corrija. A validação acontece na próxima camada.”
O erro do fornecedor é preservado no JSON. A Skill de conciliação detecta a divergência via code execution e marca como exceção. Auditoria preservada.
Princípio: uma Skill, uma responsabilidade

A nf-extractor-lumina SÓ extrai. A conciliador-lumina (próxima AP) SÓ valida e compara. A painel-divergencias-lumina (AP04) SÓ visualiza. Misturar responsabilidades numa única Skill é a receita para automações frágeis e difíceis de debugar. Na sua empresa real, essa divisão é o que diferencia automações que duram anos das que quebram no segundo mês.

O professor demonstra o fluxo completo da Skill em três momentos:

O professor abre o Project “Contas a Pagar — Lumina” criado na AP01, vai na seção de Skills, faz upload do arquivo nf-extractor-lumina.zip, e mostra a Skill aparecendo na lista de Skills ativas do Project.

Momento 2 — Testa com a NF 1 (Gellini, cenário simples)

O professor anexa a NF 1 (NF-001234-Gellini.pdf) numa nova conversa do Project e digita: “Extrai os dados dessa NF.” O Claude detecta a Skill, segue os 4 passos, e devolve o JSON estruturado em poucos segundos.

Momento 3 — Testa com a NF 2 (Embalpouch, cenário com armadilha)

O professor anexa a NF 2 (NF-005678-Embalpouch.pdf) e repete o pedido. A Skill devolve o JSON com valor_total: 12067.00. Aparentemente tudo certo. Mensagem 1 instala a Skill (zero processamento de IA, só upload). Mensagens 2 e 3 são extrações de NF — cada uma rápida. Total de 3 mensagens usadas nesta AP. Em vez de fazer “tudo em uma conversa só” e gastar mensagens por interação prolongada, você separa cada operação clara e usa o limite de forma econômica.

Demonstração
06. Anatomia da Skill — o SKILL.md em 4 passos
A estrutura que o Claude segue ao detectar a Skill

Toda Skill do Claude tem um arquivo central chamado SKILL.md que descreve o comportamento. O da nf-extractor-lumina tem 4 passos numerados:

Estrutura do SKILL.md — 4 passos

  • 1. Receber a NF — define que formatos são aceitos (PDF, imagem, XML). Se o usuário não anexou nada, a Skill pede o anexo gentilmente.
  • 2. Identificar os campos visualmente — lista 6 campos obrigatórios (número, data, razão social, CNPJ, valor total, itens) e indica onde cada um costuma aparecer no DANFE.
  • 3. Estruturar o output em JSON — schema fixo, regras de formatação (R$ vira number com ponto decimal, datas viram ISO, CNPJ preserva formatação).
  • 4. Verificação final — sanity check antes de entregar: CNPJ tem 14 dígitos, data está em ISO, valor_total é number, JSON é válido.
Por que 4 passos numerados

SKILL.md em passos numerados é seguido com alta aderência pelo Claude — muito maior que descrição em prosa corrida. A numeração serve de âncora que o modelo consulta sequencialmente. É a mesma técnica que você viu no System Prompt da AP01 (6 blocos numerados): forma é função.

07. O que observar durante a demonstração
4 sinais que diferenciam uma Skill bem instalada de um chat genérico
  • 1. Após instalar a Skill, o Project mostra uma badge ou ícone na seção Skills indicando que nf-extractor-lumina está ativa.
  • 2. Ao anexar a NF e pedir “extrai”, o Claude não pergunta detalhes nem dá instruções genéricas — ele segue direto o roteiro da Skill.
  • 3. O JSON de output é idêntico em estrutura nas duas NFs (mesmo schema, mesma ordem de campos) — apesar dos conteúdos serem diferentes. Schema fechado funcionando.
  • 4. Na NF 2, o JSON sai com valor_total: 12067.00 — a Skill não tenta corrigir o erro do fornecedor. A divergência só aparece quando o code execution roda em cima do JSON.

PASSO 07Descubra a armadilha via code execution

Este é o momento mais importante da aula. Ainda na mesma conversa, use o botão Copiar do bloco abaixo e envie a mensagem. (3 min · gasta 1 mensagem)

Instrução para descobrir a armadilha
Agora rode code execution em Python para somar os valor_unitario × quantidade de cada item do JSON da NF Embalpouch e comparar com o valor_total impresso. Mostre a soma dos itens, o valor_total impresso, e a diferença. Se houver divergência acima de R$ 1,00, alerta isso explicitamente.

Resultado esperado: o Claude executa Python, calcula a soma real dos itens (R$ 12.020,00), compara com o valor_total impresso (R$ 12.067,00) e revela uma divergência de R$ 47,00 — erro real do fornecedor que nenhum LLM detectaria só lendo o PDF.

Reflexão obrigatória

Pare por 30 segundos e pense: se a Skill tivesse “corrigido” o valor_total automaticamente para R$ 12.020,00, o departamento financeiro NUNCA SABERIA que o fornecedor errou a soma. A NF seria paga com o valor errado. Ao preservar o erro no JSON e detectá-lo só na camada de validação, você ganha rastreabilidade — pode questionar o fornecedor, pedir NF corrigida, evitar pagamento divergente.

11. Implementando na sua empresa
Adapte a Skill para extrair NFs reais do seu departamento

Adapte a Skill nf-extractor-lumina para extrair NFs da sua empresa real:

PASSO 01Baixe e descompacte a Skill

Baixe a Skill nf-extractor-lumina.zip e descompacte. Você vai editar o arquivo SKILL.md interno.

PASSO 02Adapte o schema JSON se sua empresa precisa de campos adicionais
  • Sua operação trabalha com retenção de impostos (IR, INSS, CSLL)? Adicione campos retencoes ao JSON
  • Suas NFs vêm com chave de acesso da NF-e em destaque? Adicione campo chave_acesso
  • Você usa centro de custo? Adicione campo centro_custo (mas mantenha como null quando não vier preenchido na NF)
Regra de ouro

Todo campo adicionado precisa ser CONSUMIDO por uma Skill a jusante. Se ninguém vai usar, não adicione.

PASSO 03Adapte os passos para o formato específico da sua empresa

Se sua empresa recebe NFs em formato muito específico (NFS-e de prefeitura X, conhecimento de transporte, etc.), adicione menções específicas no Passo 2 da Skill sobre onde os campos costumam aparecer.

PASSO 04Recompacte em .zip

Recompacte com o nome nf-extractor-[sua-empresa].zip.

PASSO 05Instale no Project real da sua empresa

Instale no Project real que você criou no Desafio da AP01 (o Project Contas a Pagar da sua empresa).

PASSO 06Teste com 3 NFs reais — uma simples, uma complexa, uma com anomalia

Teste com 3 NFs reais do seu departamento — uma simples, uma complexa, e uma que você sabe que tem alguma anomalia. Para cada uma, repita o teste de code execution que você fez na NF 2 da Lumina (somar itens vs valor_total impresso).

Anote quantas das 3 NFs reais têm divergência real.

Resultado esperado

Skill de extração instalada no Project real da sua empresa, validada com 3 NFs reais, e descoberta empírica de quantas NFs reais da sua empresa têm divergências de soma que passavam batido antes (geralmente é assustadoramente alto — entre 5% e 15% das NFs em operações sem revisão sistemática).

Tempo estimado para o desafio

30 a 45 minutos.

12. Conexão com o programa
O impacto financeiro real de detectar divergências automaticamente
Por que descobrir divergências no fornecedor importa mais do que parece

Numa operação típica de 200 NFs/mês com 10% de divergência não detectada, e ticket médio de R$ 8.000, você está pagando aproximadamente R$ 160.000 a mais por mês que não deveria — só porque ninguém reconfere centavo das NFs. Em 12 meses, isso é R$ 1,92 milhão. A Skill que você instalou hoje, combinada com a próxima de conciliação, é o ponto onde esse vazamento para. É por isso que a AP03 vem a seguir.
