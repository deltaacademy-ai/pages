Introdução
03. Os 4 arquivos de contexto explicados
Plano de Fornecedores vs Política de Conciliação — e os 2 arquivos complementares

Os 3 primeiros elementos do Project são textos simples — fáceis de entender. O quarto elemento (arquivos de contexto) merece explicação maior, porque dois arquivos parecem similares mas têm papéis bem diferentes:

Plano de Fornecedores
plano-fornecedores-lumina.md
É a lista estruturada de todos os fornecedores que a Lumina paga, organizada por categoria (matéria-prima, embalagem, frete/logística, marketing, administrativo). Pense como o catálogo de fornecedores da empresa — mostra quem são os fornecedores e em que categoria cada um se encaixa.
Política de Conciliação
politica-conciliacao-lumina.md
É o manual de regras que diz como decidir se uma NF bate com o pedido quando há diferenças de valor, prazo ou quantidade. Pense como o livro de regras do auditor — mostra as decisões difíceis e como resolvê-las.
Por que separar em dois arquivos

A Skill da AP03 (conciliação) consulta os dois ao mesmo tempo: o plano de fornecedores identifica de quem é a NF e em que categoria classificar, e a política de conciliação decide se o valor da NF bate com o pedido dentro da tolerância aceita pela Lumina. Sem o segundo, o Claude inventa critérios próprios — comprometendo a consistência do fechamento mensal.

Identidade Visual — identidade-visual-lumina.md

Define a estética minimalismo wellness premium da Lumina: paleta de cores (branco quente, verde-musgo, coral terroso), tipografia (Fraunces e Inter), princípios de design. Será usada principalmente na AP04 quando a Skill gerar o painel HTML de divergências.

Histórico de Pedidos — historico-pedidos-lumina.xlsx

Planilha leve com 1 aba contendo ~30 pedidos do mês vigente que a Lumina fez aos seus fornecedores recorrentes. Cada linha tem: número do pedido, fornecedor, CNPJ, categoria, data de emissão, valor esperado, prazo de pagamento, status.

Esse arquivo é a fonte de verdade que a Skill da AP03 vai consultar para cruzar com as NFs recebidas e detectar divergências.

04. Conceito — LLM lê, código calcula
A regra que separa quem automatiza bem de quem não automatiza

Quando você pede para um LLM “calcular o valor total da NF e comparar com o pedido”, ele faz isso por aproximação de tokens — não por aritmética real. Em valores baixos costuma acertar, mas em centavos críticos ele pode errar a terceira casa decimal, arredondar silenciosamente, ou inventar um total plausível mas errado.

A regra que separa quem automatiza bem de quem não automatiza: LLM extrai e estrutura, código calcula e compara.

Pedir pro LLM calcular
“Some os itens dessa NF e me diga se bate com o pedido.”
O LLM responde uma soma plausível, mas pode errar centavos. Em conciliação financeira, centavo errado = divergência falsa ou pagamento duplicado.
Dividir o trabalho
“Extraia os itens da NF em JSON estruturado. Depois rode code execution para somar e comparar.”
O LLM faz o que ele faz bem (leitura e estruturação). O código faz o que ele faz bem (aritmética exata). Resultado: zero erro de cálculo.
Analogia do escriturário e do contador

É a diferença entre um escriturário que lê documentos manuscritos e organiza fichas, e um contador que faz as contas baterem no centavo. Um lê. O outro calcula. Misturar os dois papéis é o erro clássico de quem começa a automatizar com IA. No Project Contas a Pagar Lumina, o Claude é o escriturário (lê NFs, extrai campos, estrutura em JSON) e o code execution é o contador (soma, compara, valida tolerâncias).

06. Anatomia de um System Prompt de Project
Estrutura em 5 blocos numerados que será carregada nas Instruções do Project

O elemento mais rico dos 4 é a Instrução do Project (o System Prompt). É ele que define o “comportamento” do assistente em todas as conversas dentro do Project. Veja a estrutura usada no System Prompt da Lumina que você vai copiar daqui a pouco:

Estrutura do System Prompt — 5 blocos

  • 1. Identidade — papel do assistente (analista de contas a pagar sênior da Lumina) + tom (direto, preciso, wellness premium).
  • 2. Contexto da empresa — wellness em crescimento, 3 produtos de bala de goma funcional, canal online + 2 lojas físicas em SP e RJ, ~200 NFs/mês.
  • 3. Diretrizes de extração de NF — sempre JSON estruturado, nunca calcular sem code execution, sempre validar CNPJ contra o plano.
  • 4. Diretrizes de conciliação — tolerância de 1%, escalada acima de R$ 5.000, detecção de duplicidade por CNPJ + número da NF + valor.
  • 5. Formato de output — JSON para extração, tabela para conciliação, HTML para painel.
Por que esse formato

System Prompts em blocos numerados (1. Identidade, 2. Contexto, etc.) são consistentemente seguidos pelo Claude em conversas longas. Bullet points dispersos ou parágrafos corridos têm aderência menor — a estrutura numerada serve de “âncora” que o modelo consulta a cada nova mensagem.

Texto para o Campo 3 — Instruções do Project (System Prompt completo)
# 1. IDENTIDADE Você é o assistente de Contas a Pagar da Lumina, atuando como analista financeiro sênior do departamento. Seu tom é direto, preciso e alinhado à energia wellness premium da marca: profissional sem rigidez, técnico sem jargão desnecessário. # 2. CONTEXTO DA EMPRESA A Lumina é uma empresa em crescimento no segmento de wellness premium. Vende balas de goma funcionais para pessoas que praticam exercícios físicos. Portfólio: 3 produtos (bala pré-treino, bala com creatina, bala com whey protein). Canais de venda: e-commerce próprio (canal principal) + 2 lojas físicas próprias (uma em São Paulo, uma no Rio de Janeiro). Volume de notas fiscais a pagar: aproximadamente 200 NFs/mês. Fornecedores divididos em 5 categorias: Matéria-Prima, Embalagem, Frete e Logística, Marketing, Administrativo. # 3. DIRETRIZES DE EXTRAÇÃO DE NF - Sempre extraia os campos da NF em JSON estruturado, com tipos explícitos (string para CNPJ e número da NF, number para valores monetários, date no formato ISO para datas). - NUNCA calcule o valor total a partir dos itens individuais sem usar code execution. LLM não soma centavo confiavelmente. - Sempre valide o CNPJ extraído contra o arquivo plano-fornecedores-lumina.md antes de prosseguir. Se o CNPJ não estiver cadastrado, marque a NF como "Fornecedor não cadastrado". - Para NFs em PDF ou imagem, use a capacidade de visão do Claude. Para NFs em XML, extraia diretamente dos campos estruturados. # 4. DIRETRIZES DE CONCILIAÇÃO - Tolerância de valor: até 1% de diferença entre NF e pedido = conciliado automaticamente. Acima disso, marque como exceção. - Tolerância de prazo: até 5 dias úteis após a data prevista no pedido = OK. Acima disso, marque como exceção de prazo. - Escalada urgente: divergência acima de R$ 5.000 em valor absoluto vai para a controller, prioridade alta. - Detecção de duplicidade: combinação idêntica de CNPJ + número da NF + valor (tolerância R$ 0,01) em janela de 30 dias = bloqueio automático. - Toda comparação numérica é feita via code execution, nunca por estimativa textual. # 5. FORMATO DE OUTPUT - Extração de NF: JSON estruturado (puro, sem markdown ao redor), pronto para consumo programático. - Relatório de conciliação: tabela markdown com colunas Nº NF, Fornecedor, Valor NF, Valor Pedido, Status, Categoria. - Painel de divergências: HTML autocontido com paleta da Lumina (consultar identidade-visual-lumina.md), pronto para apresentação à liderança. - Mensagens conversacionais: prosa direta, sem florear, com bullets quando há mais de 3 itens. # 6. FERRAMENTAS E SKILLS DESTE PROJECT Este Project usará as seguintes capacidades nas APs seguintes: - Visão do Claude (leitura de NFs em PDF e imagem) - Code execution (cálculos, somas, validações numéricas) - Skills customizadas (extração, conciliação, painel) - Tarefas agendadas via Cowork (conciliação semanal automática)

PASSO 01Adapte os 3 textos pré-prontos

Baixe os arquivos da Lumina como base e substitua o conteúdo pelos dados reais da sua empresa:

  • plano-fornecedores-[sua-empresa].md — preencha com os fornecedores reais cadastrados, organizados nas categorias da sua empresa
  • politica-conciliacao-[sua-empresa].md — substitua as regras pelas políticas reais (tolerâncias, escalas, prazos de pagamento)

PASSO 05Valide com 1 fornecedor real

Abra o Project e peça ao Claude para identificar a categoria de 1 fornecedor real consultando o plano que você acabou de carregar.

45 a 60 minutos — a maior parte é adaptar o plano de fornecedores e exportar o histórico de pedidos real do seu ERP.
